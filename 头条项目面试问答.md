## 一、项目架构与设计

### 1. 请介绍一下这个头条项目的整体架构

**答案**：这是一个基于 FastAPI 的新闻资讯后端系统，采用经典的三层架构：

- **路由层（Routers）**：拆分 users、news、favorite、history、chat 5 个业务路由，共 20 个 API 接口
- **业务层（CRUD/Services）**：封装数据库操作和业务逻辑
- **数据层（Models）**：使用 SQLAlchemy Async 定义 User、News、Favorite、History 四个数据模型

技术栈包括：FastAPI + MySQL + Redis + SQLAlchemy Async + LangGraph + Chroma + Celery，实现了用户注册登录、新闻浏览、收藏、历史记录和智能问答等核心功能。

### 2. 为什么选择 FastAPI 而不是 Django 或 Flask？

**答案**：选择 FastAPI 主要基于以下考虑：

1. **原生异步支持**：项目大量使用 async/await 进行数据库查询、缓存访问和 HTTP 请求，FastAPI 原生支持异步，性能更优
2. **自动文档生成**：OpenAPI 文档自动生成，便于前后端协作
3. **类型校验**：基于 Pydantic 的请求/响应校验，开发体验好，错误更早暴露
4. **轻量灵活**：相比 Django 更轻量，相比 Flask 有更好的异步和类型支持

### 3. 项目中如何管理数据库连接的生命周期？

**答案**：使用 SQLAlchemy Async 的 `AsyncSessionLocal` 会话工厂，通过 `async with` 上下文管理器确保会话正确关闭：

```python
async with AsyncSessionLocal() as db:
    # 执行数据库操作
    result = await db.execute(stmt)
```

这种方式保证无论正常执行还是异常，连接都会被正确释放，避免连接泄漏。

## 二、多级缓存设计

### 4. 请详细介绍一下多级缓存架构的设计

**答案**：设计了 L1 本地缓存 + L2 Redis + DB 回源的三级缓存架构：

- **L1 本地缓存**：进程内 LRU + TTL 缓存，默认容量 1000，TTL 300 秒，延迟极低
- **L2 Redis**：跨进程共享缓存，支持分布式场景下的缓存复用
- **DB 回源**：最终数据源

读取顺序：L1 → L2 → DB，L2 命中后回填 L1，DB 回源后同时填充两级缓存。

### 5. 什么是缓存穿透？项目中如何解决？

**答案**：缓存穿透指查询不存在的数据时，请求直接穿透缓存打到数据库。

项目中通过**空值缓存（负缓存）**解决：当数据库查询返回 None 或空集合时，也将这个"空结果"写入缓存，TTL 设置为 300 秒：

```python
if db_value is None:
    await self.l2.set(key, None, ex=NULL_CACHE_EXPIRE_SECONDS)
    self.l1.set(key, None, ttl=NULL_CACHE_EXPIRE_SECONDS)
```

### 6. 什么是缓存击穿？项目中如何解决？

**答案**：缓存击穿指热点 key 过期瞬间，大量并发请求同时穿透到数据库。

项目中通过 **SingleFlight** 模式解决：相同 key 的并发请求会被合并，只执行一次数据库查询，其他请求等待并复用结果：

```python
return await singleflight.do(f"multi-cache:{key}", load_and_fill)
```

### 7. 什么是缓存雪崩？项目中如何解决？

**答案**：缓存雪崩指大量缓存同时过期，导致数据库瞬时压力剧增。

项目中通过 **TTL 随机抖动** 解决：在基础过期时间上增加随机偏移，避免大量 key 同时过期：

```python
def _ttl_with_jitter(expire: int) -> int:
    jitter = min(max(expire // 10, 1), RedisConfig.CACHE_RANDOM_OFFSET)
    return expire + random.randint(0, jitter)
```

### 8. 什么是 stale-while-revalidate 策略？在什么场景使用？

**答案**：stale-while-revalidate 是一种热点缓存策略，允许在缓存"软过期"后先返回旧值，同时异步刷新缓存。

项目中通过 `logic_cache` 装饰器实现，适用于：

- 分类列表、配置数据等"读多写少"的数据
- 允许短暂旧值但需要后台更新的场景

实现原理：缓存写入时记录 `expire_at`（软过期）和 `hard_expire`（硬过期），读取时：

- 软过期未到：直接返回
- 软过期已到但硬过期未到：返回旧值 + 触发异步刷新
- 硬过期已到：同步回源

## 三、限流设计

### 9. 请介绍一下项目的限流架构

**答案**：项目实现了**统一限流入口** `UnifiedRateLimiter`，采用三层防护：

1. **黑名单检测**：直接拦截被封禁的用户/IP
2. **滑动窗口防刷**：检测短时高频请求，超频自动拉黑
3. **令牌桶限流**：平滑限流，允许一定突发流量

限流维度支持：IP 维度、用户 ID 维度、组合维度（user_id + IP）。

### 10. 为什么选择令牌桶算法？它的原理是什么？

**答案**：令牌桶算法适合 Web 限流场景，因为：

- 允许一定程度的突发流量（桶内有存量令牌时）
- 长期来看仍能限制平均速率

原理：系统以固定速率往桶里放令牌，请求到达时从桶里取令牌，有令牌则通过，无令牌则拒绝或等待。项目中桶容量和补充速率可配置。

### 11. Redis 限流如何保证原子性？

**答案**：使用 **Lua 脚本** 在 Redis 侧原子执行令牌补充和扣减：

```lua
local tokens = tonumber(data[1]) or capacity
local elapsed = (now_ms - last_time) / 1000
tokens = math.min(capacity, tokens + elapsed * rate)
-- 扣减逻辑
if tokens >= 1 then
    tokens = tokens - 1
end
```

Lua 脚本在 Redis 中原子执行，避免多客户端竞态条件。

### 12. Redis 不可用时限流如何降级？

**答案**：项目中实现了**本地令牌桶降级**机制：

```python
result = await RedisRateLimit.token_limit(identity, self.cfg)
if "Redis异常" in result.reason and settings.ENABLE_LOCAL_FALLBACK:
    bucket = await LocalTokenBucket.get_instance(identity.key, ...)
    result = await bucket.try_consume()
```

当 Redis 限流失败时，自动切换到进程内本地令牌桶，保证限流能力不中断。

### 13. 本地令牌桶如何防止内存泄漏？

**答案**：本地令牌桶使用 LRU 策略清理旧实例：

```python
if len(cls._instances) >= cls._max_instances:  # 默认 10000
    keys_to_remove = list(cls._instances.keys())[:cls._max_instances // 2]
    for k in keys_to_remove:
        del cls._instances[k]
```

当实例数超过阈值时，清理最旧的一半实例。

## 四、RAG 与智能问答

### 14. 请介绍一下新闻问答 Agent 的设计

**答案**：基于 LangGraph 构建了 analyze → retrieve → generate 三节点工作流：

1. **analyze 节点**：分析用户问题，提取关键词和分类
2. **retrieve 节点**：根据分析结果进行向量检索
3. **generate 节点**：基于检索结果生成回答

支持最多 3 轮重试闭环，当回答不满意时自动重新分析检索。

### 15. LangGraph 工作流是如何编排的？

**答案**：使用 `StateGraph` 定义工作流，通过 `TypedDict` 声明全局状态：

```python
workflow = StateGraph(AgentState)
workflow.add_node("analyze", analyze_node)
workflow.add_node("retrieve", retrieve_node)
workflow.add_node("generate", generate_node)
workflow.set_entry_point("analyze")
workflow.add_edge("analyze", "retrieve")
workflow.add_edge("retrieve", "generate")
workflow.add_conditional_edges("generate", should_continue, {"end": END, "retry": "analyze"})
```

每个节点返回的 dict 会自动合并到全局状态中。

### 16. 向量检索是如何实现的？

**答案**：使用 Chroma 向量库 + HuggingFace Embedding：

- **Embedding 模型**：`paraphrase-multilingual-MiniLM-L12-v2`，384 维，支持多语言
- **向量存储**：Chroma 持久化到本地 `data/chroma` 目录
- **检索流程**：用户问题 → 向量化 → 相似度检索 → 返回 top-k 文档

支持按分类过滤：`filter_dict = {"category_id": category_id}`

### 17. 如何保证向量检索结果与业务数据一致？

**答案**：采用**检索后回查 MySQL** 策略：

向量库只存储新闻 ID、标题、摘要等元数据，检索后根据 `news_id` 回查 MySQL 获取完整新闻详情。这样即使 MySQL 中的新闻内容更新，检索结果也能保持最新。

### 18. Embedding 模型加载会影响服务启动吗？

**答案**：采用**延迟加载（懒加载）**策略，避免服务启动时阻塞：

```python
async def aget_embeddings(self) -> HuggingFaceEmbeddings:
    if self._instance is not None:
        return self._instance
    await run_in_threadpool(self.get_embeddings)  # 在线程池中加载
    return self.get_embeddings()
```

同时在 `main.py` 的 `lifespan` 中可选预热 RAG 资源。

### 19. 索引构建支持哪些模式？

**答案**：`indexer.py` 支持三种索引模式：

1. **全量索引**：`index_all_news()` - 分批读取所有新闻导入向量库
2. **增量索引**：`index_news_by_id()` / `index_news_by_ids()` - 只索引指定新闻
3. **重建索引**：`reindex_all()` - 清空向量库后重新导入

## 五、异步任务调度

### 20. Celery 在项目中的作用是什么？

**答案**：Celery 用于处理耗时操作，避免阻塞主服务：

1. **新闻抓取**：定时从新浪、腾讯抓取新闻，每 6 小时执行一次
2. **热度统计**：定期计算新闻热度，更新排行榜，每小时执行一次
3. **数据同步**：批量数据处理和迁移

### 21. Celery 任务队列是如何设计的？

**答案**：按业务拆分队列，实现资源隔离：

- **default 队列**：默认队列，处理一般任务
- **news 队列**：新闻相关任务（抓取、索引）
- **statistics 队列**：统计相关任务（热度计算、排行榜）

路由规则：`tasks.news_tasks.*` → news 队列，`tasks.statistics_tasks.*` → statistics 队列。

### 22. Celery 任务如何保证可靠性？

**答案**：通过以下配置保证任务不丢失：

- **ACKS_LATE**：任务执行成功后才确认，Worker 崩溃时任务会被重新执行
- **PREFETCH_MULTIPLIER**：控制 Worker 预取任务数，避免任务分配不均
- **max_retries**：任务失败最多重试 3 次
- **软/硬超时**：软超时 5 分钟（可捕获清理），硬超时 10 分钟（强制终止）

### 23. Celery 任务中如何执行异步代码？

**答案**：Celery 任务函数是同步的，需要在任务内部创建事件循环：

```python
def fetch_and_save_news(self):
    async def _execute():
        # 异步逻辑
        async with AsyncSessionLocal() as db:
            news_list = await NewsSpiderService.fetch_all_news()
  
    loop = asyncio.new_event_loop()
    asyncio.set_event_loop(loop)
    try:
        return loop.run_until_complete(_execute())
    finally:
        loop.close()
```

### 25. 新闻分类是如何实现的？

**答案**：基于关键词匹配的分类算法：

```python
def classify_news(title: str, content: str = "", description: str = "") -> int:
    text = f"{title} {content} {description}".lower()
    category_scores = {}
    for category, keywords in settings.SPIDER_CLASSIFICATION_RULES.items():
        score = sum(1 for keyword in keywords if keyword.lower() in text)
        if score > 0:
            category_scores[category] = score
    best_category = max(category_scores, key=category_scores.get)
    return category_id_map.get(best_category, default_category_id)
```

统计标题、内容、描述中各分类关键词出现次数，选择得分最高的分类。

## 七、并发与性能

### 26. SingleFlight 的实现原理是什么？

**答案**：SingleFlight 合并相同 key 的并发请求：

```python
async def do(self, key: str, func: Callable) -> Any:
    if key in self._inflight:
        return await self._inflight[key]  # 复用已有的 Future
  
    future = asyncio.Future()
    self._inflight[key] = future
    try:
        result = await func()
        future.set_result(result)
        return result
    finally:
        self._inflight.pop(key, None)
```

使用 `asyncio.Future` 让后续请求等待第一个请求的结果。

### 27. 本地缓存如何保证线程安全？

**答案**：使用 `RLock` 可重入锁保护所有读写操作：

```python
def get_entry(self, key: Any) -> LocalCacheEntry:
    with self._lock:
        self._purge_expired_locked()
        stored = self._cache.get(key)
        # ...
```

同时使用 `OrderedDict` 维护 LRU 顺序，访问时 `move_to_end` 更新顺序。

### 28. LangGraph 工作流如何保证并发安全？

**答案**：使用 `asyncio.Lock` 双重检查锁保证工作流只构建一次：

```python
async def _get_graph(self):
    if self._graph is not None:
        return self._graph
    async with self._build_lock:
        if self._graph is None:
            self._graph = self._build_graph_sync()
    return self._graph
```

## 八、工程实践

### 29. 项目中如何处理异常？

**答案**：

1. **全局异常处理器**：`register_exception_handlers(app)` 注册统一异常处理
2. **限流异常**：单独注册 `RateLimitExceeded` 处理器，返回 429 状态码
3. **业务异常**：在路由层捕获并转换为 HTTPException
4. **降级策略**：缓存/限流组件异常时降级放行，优先保障服务可用性

### 30. 项目中如何实现单例模式？

**答案**：主要使用两种方式：

1. **模块级变量 + getter 函数**：

```python
_agent_runner: NewsQaAgentRunner | None = None
def get_agent_runner() -> NewsQaAgentRunner:
    global _agent_runner
    if _agent_runner is None:
        _agent_runner = NewsQaAgentRunner()
    return _agent_runner
```

2. **`__new__` + 类级锁**：

```python
class MultiLevelCache:
    _instance_lock: RLock = RLock()
    _instance: Optional["MultiLevelCache"] = None
  
    def __new__(cls, *args, **kwargs) -> "MultiLevelCache":
        with cls._instance_lock:
            if cls._instance is None:
                cls._instance = super().__new__(cls)
        return cls._instance
```
